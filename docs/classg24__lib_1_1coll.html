Detailed Description

template<class Type, class indexer = g24_lib::Definitions::default_unsigned_indexer>
class g24_lib::coll< Type, indexer >

Holds a run-time defined number of objects in a run-time defined number of dimensions.

Template Parameters
TypeThe type of the elements of the array.
indexerThe type that should be able to hold the size of the array.
Precondition
indexer must satisfy the constraints placed on the choice of g24_lib::Definitions::default_unsigned_indexer.

Suppose we want to store something in n+1 dimensions, with k_0, k_1, ... , k_n elements in each dimension.
g24_lib::coll will allocate an array of k_0 * k_1 * ... * k_n elements. The elements at each dimension will be stored at decreasing strides, much in the same way as native multidimensional static arrays (such as int x[10][11][12]).

The strides between each element in a given dimension will be: k_1 * k_2 * ... * k_n, k_2 * ... * k_n, ... , k_n , 1 (that is, in the last dimension the elements are stored contiguosly).

Deprecated:
After seeing much use during the projects that originated the first versions of this library, the authors eventually concluded that the run-time defined number of dimensions was not a significant advantage, quite the opposite, since it forced the usage of the (also deprecated) g24_lib::point class and more dynamic allocations than desireable. A more reasonable alternative is to use g24_lib::ndview to specify a run-time number of elements in a compile-time number of dimensions and couple this with any array (not necessarily g24_lib::simple_array, though that would be the most natural choice for this library...) using g24_lib::ndview::array_size() to get the necessary number of elements and access ordered positions through g24_lib::ndview.

◆ to_elem()

template<class Type , class indexer = g24_lib::Definitions::default_unsigned_indexer>
indexer g24_lib::coll< Type, indexer >::to_elem ( const point< indexer, indexer > &  p) const
inline

Converts from an ordered position to an index.

Remarks
In essence, this is the dot product between the ordered position (understood as a vector) and the stride at each dimension (as a vector as well).
To understand the algorithm, take for example a 3d collection of elements with {11, 12, 13} in each dimension. The strides will be {156,13,1}.
Suppose one wants to find the index that corresponds to the position {4,5,6}. Given the way positions are translated into indexes, one needs to compute:
4 * 156 + 5 * 13 + 1 * 6 = 624 + 65 + 6 = 695.
See also

Converts from an index to an ordered position.

Remarks
To understand the algorithm, take for example a 3d collection of elements with {11, 12, 13} in each dimension. The strides will be {156,13,1}.
Suppose one wants to find the ordered position that corresponds to the index 123. Given the way positions are translated into indexes, one needs to compute:
#- 123 / 156 = 0 (in integer arithmetic), 123 % 156 = 123: The first component of the position is 0, we proceed with the number 123. #- 123 / 13 = 9, 123 % 13 = 6: The second component of the position is 9, we proceed with the number 6 #- 6 / 1 = 6: The third and last component of the position is 6.
See also
g24_lib::coll::to_elem
